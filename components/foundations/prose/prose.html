p · 14px / 400 / base leading / max-width 65ch

This is a default paragraph. It uses the regular weight and the base line height. The maximum width is constrained to 65 characters so that long lines do not hurt readability. If a paragraph reaches this width, it will wrap rather than stretch to fill its container.

This is a second paragraph. The space between paragraphs comes from the bottom margin of the first one. The top of a paragraph block has no extra space above it, and the last paragraph in a container has no bottom margin.

inline emphasis

This paragraph contains strong emphasis using the medium weight, and emphatic italic for tonal stress. Inline small text is one step smaller than body. None of these use a bold weight — the system has only regular and medium.

article · paragraphs get loose leading

This paragraph is inside an <article> element. Paragraphs inside articles get looser line height (1.7 instead of 1.55) because they are intended for sustained reading rather than UI consumption.

The opt-in is semantic. Wrap a region of long-form prose in <article> and the leading adjusts automatically. No class needed. This is the foundation's preferred pattern, but a .prose-long class is also available for cases where <article> is not appropriate.

composition — all elements in real use

Designing for the edges

Most design systems optimise for the centre: fast connections, capable devices, design-literate users. seedbank.design optimises for the edges — the rural broadband connection, the cheap Android phone, the e-ink reader, the printout pinned to a community board.

What that means in practice

It means graceful degradation is the brief, not an afterthought. A component that depends on CSS to communicate meaning has failed. A button that's only a button when the icon font loads has failed. A status indicator that relies on colour alone has failed.

It also means resisting the temptation to add visual complexity that doesn't survive being clone-and-adapted by a community in conditions you can't foresee.

Concrete implications

Every component is tested against three criteria: renders correctly without CSS loaded, works at 200% zoom, and works on a 320px viewport. These aren't ceilings — they're floors. Other concerns (keyboard navigation, screen reader behaviour, print rendering, dark mode) matter too, but the three above are the standing checklist every component faces.

A note on print

The system is tested in print mode because communities frequently print what they organise around — meeting agendas, repair guides, mutual aid forms, event notices. A design system that doesn't survive contact with a printer is not serving these communities.

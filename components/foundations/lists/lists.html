Concrete implications of the edges principle

Designing for the edges shapes every component decision. The visible consequences fall into a few clear categories:

  • No gradients, no decorative effects, no font weights that disappear on cheap screens. The system uses only regular and medium weights for this reason.
  • Every component is tested in print mode and at low bandwidth before release. The acceptance bar is documented in TESTING.md.
  • Icons always paired with visible labels — never icon-only. Pure-icon controls fail when the icon font doesn't load or the user doesn't recognise the symbol.
  • States are explicit and named, not implied by colour alone. Colour reinforces meaning; it doesn't carry it on its own.

Each of these reads as a constraint, but in practice they tend to make components simpler, not more elaborate. The discipline of designing for the edges removes the temptation to add visual complexity that wouldn't survive being clone-and-adapted by a community in conditions we can't foresee.

What the three-criterion test catches

  1. Renders correctly without CSS loaded — catches components that depend on CSS to communicate meaning.
  2. Works at 200% zoom — catches fixed-pixel layouts and components that break under reflow.
  3. Works on a 320px viewport — catches assumptions about horizontal space.

when not to use a list

Avoid — single-item list

A list with one item is not a list. If the content has only one element, write it as a paragraph. Screen readers announce the list and its single item, which is noise.

Avoid — list as layout grid

Don't reach for <ul> to lay out cards, navigation items, or grids of content. Use semantic containers — <nav> for navigation, <article> or <section> for content groupings — and grid or flex layout for the visual arrangement. Lists are for content that reads as a list.

Avoid — bordered rows via <li>

Status rows, card lists, and other UI patterns that use list-like markup with borders, left-rules, or other treatments belong in the patterns layer. This foundation styles content lists only. Patterns will reset the foundation's styling on the elements they use.

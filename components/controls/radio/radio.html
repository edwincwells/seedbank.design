when not to use a radio group

Avoid — a single radio

A radio with nothing to be exclusive with is a checkbox in a confusing costume. Once selected, it can't be unselected — so it can't represent a "yes / no" toggle. If you have one option, use a checkbox. If you have a binary choice, use two radios in a group (with a legend that names the question).

Avoid — radios without a <legend>

The <legend> provides the accessible name for the group. Without it, screen readers announce each radio individually with no context — "Email, radio button, not checked" tells the user nothing about what question they're answering. The legend is the group's question; the radios are its options. Both are required.

Avoid — radios where the user can pick zero or more than one

Radios mean "select exactly one." If zero is a valid answer for an optional group, include an explicit "None" option (see the dietary example above) — don't make the user guess that they can skip the question. If more than one is valid, use checkboxes. Picking the wrong control sets the wrong expectation about what's allowed.

Avoid — fighting the native keyboard model

The browser's default behaviour for a radio group is: Tab moves focus past the group; arrow keys move within the group and change selection, skipping disabled radios. Don't add tabindex="0" to each radio (it breaks the group navigation), and don't add JS to "fix" the arrow-key behaviour (it already works). The native model is correct; trust it.

Avoid — marking pristine required radios invalid

Don't apply aria-invalid="true" to a required radio group on page load. Marking every vote option invalid before the user has had a chance to interact makes the whole form look broken. Apply aria-invalid after a validation attempt — typically on form submit — and apply it to every radio in the failing group, not just one. The visual signal (red borders across the group) reads the group as a whole.

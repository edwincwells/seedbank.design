when not to use a checkbox

Avoid — no label

Every checkbox needs an accessible name. Either wrap it in a <label> as shown above, associate a separate <label for="..."> with the input's id, or provide an aria-label. An unlabelled checkbox is invisible to screen readers and ambiguous for everyone else.

Avoid — checkbox when a toggle would be clearer

Use a checkbox when the user is selecting from a set of options or confirming a stateful choice (subscribe / agree / include). Don't use a checkbox for binary on/off settings where the action takes effect immediately — that's a toggle switch, which behaves differently (immediate apply vs queued change) and reads differently to users.

Avoid — checkbox group where the user must pick exactly one

If the user must select exactly one option from a list (and only one), use radio buttons. Checkboxes imply "select zero or more"; radios imply "select exactly one." Picking the wrong control sets the wrong expectation about what's allowed.

Avoid — marking pristine required checkboxes invalid

Don't apply aria-invalid="true" to a required checkbox on page load. Marking the "I agree" checkbox invalid before the user has had a chance to interact with the form makes the whole form look broken. Apply aria-invalid after a validation attempt — typically on form submit.

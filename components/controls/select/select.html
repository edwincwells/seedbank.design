Wrong-use blocks

don't: use <select> for fewer than ~5 options

A short list of mutually exclusive options is better served by a radio group — every option is visible without an interaction, and the choice is immediate. Selects hide their options behind a panel that costs a click and a moment of cognitive load.

Use <select> when the list is long enough that showing all options inline would dominate the form (regions, currencies, time zones, languages). Use radios when the user benefits from seeing all options at once.

don't: rely on <optgroup> for visually meaningful grouping

The grouping is semantic — assistive tech announces it — but the visual rendering is set by the OS and varies. If users need to see the groups clearly (e.g. for a long sortable list), a custom listbox or a different control is the right answer. <optgroup> is good for accessibility metadata; it's not a reliable visual structure.

don't: assume the open dropdown panel can be styled

The panel that opens when a user clicks the select is rendered by the operating system. CSS cannot reach it. The closed state matches the system's vocabulary; the open state will look like whatever macOS, Windows, iOS, or Android draws. This is a documented limitation, not a bug — honest about what the platform allows.

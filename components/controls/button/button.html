button

The primary interactive control of the system. Four variants by emphasis, two sizes, plus loading and disabled states. The default <button> with no class is the primary variant — no class lookup needed for the common case.

Buttons inside forms — read this

HTML has a quirk worth knowing about. When you put a <button> inside a <form>, the browser assumes its type is submit unless you say otherwise. A button labelled "Show password" or "Add row" will submit the form when clicked — and pressing Enter inside a text input will also click whichever submit button comes first in the markup.

The fix is to set type="button" explicitly on every button inside a form that isn't meant to submit:

correct — explicit types on every button in a form

Being explicit about type="submit" on the actual submit button isn't required (it's the default) but it makes the intent legible. The type="button" on the "Show password" button is what prevents the bug.

when not to use a button

Avoid — icon-only buttons

Every button needs a visible text label. An icon alone — even a familiar one like a cog or a trash can — fails when the icon font doesn't load, when the user doesn't recognise the symbol, or when the page is printed. If horizontal space is tight, use a smaller size (.btn-sm) before dropping the label.

If your interface genuinely needs an icon-only control for space reasons (a close button in a toolbar), provide an aria-label — but consider whether the affordance is clear enough for non-expert users first.

Avoid — <div> or <span> as a button

If it acts like a button, it should be a <button> (or an <a> if the action navigates). Using <div onclick="..."> makes the control invisible to keyboard users and screen readers. The native element handles focus, keyboard activation (Space and Enter), and assistive technology announcement for free.

Avoid — ambiguous labels like "Click here" or "Submit"

Labels should name the action: "Save changes", "Sign in", "Add member", "Delete record". A label of "Submit" or "OK" forces the user to read surrounding context to understand what will happen. This is especially important for screen reader users, who often navigate by tabbing through controls — each label needs to make sense on its own.

Avoid — <a class="btn"> for non-navigation actions

If the action submits a form, triggers JS, or does anything other than navigate to a new URL, use a <button>. Links and buttons have different keyboard behaviour (Enter activates both; Space only activates buttons) and different screen reader announcements. Styling a link as a button doesn't change its semantics.
